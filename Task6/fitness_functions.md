# Задание 6. Проектируем fitness-функции

Ниже представлена таблица с описанием архитектурных фитнес-функций, спроектированных для платежной системы согласно требованиям.

| Название функции                | Назначение                                           | Что проверяется                                                                                                                                                         | Действие при нарушении                                                                                                    | Способ реализации                                                                                                                                |
|:--------------------------------|:-----------------------------------------------------|:------------------------------------------------------------------------------------------------------------------------------------------------------------------------|:--------------------------------------------------------------------------------------------------------------------------|:-------------------------------------------------------------------------------------------------------------------------------------------------|
| **SagaStateConsistency**        | Консистентность состояний Saga                       | У всех Saga есть корректные конечные состояния (отсутствуют транзакции, "брошенные" на полпути)                                                                         | Восстановление состояния или откат до предыдущего стабильного состояния (запуск компенсирующей транзакции)                | Scheduled job, который проверяет записи в таблице `saga_states` (или запрашивает API оркестратора) на наличие «зависших» незавершенных состояний |
| **PaymentTimeoutControl**       | Контроль времени выполнения платежей                 | Платежи в статусе "PROCESSING" дольше 30 минут (или дольше SLA на ручную проверку)                                                                                      | Автоматический возврат средств (Refund) и уведомление клиента                                                             | Cron job, который ищет в БД платежи с timestamp старше 30 минут и статусом PROCESSING                                                            |
| **CrossServiceDataConsistency** | Проверка консистентности данных между микросервисами | Совпадение сумм успешных списаний (Payment Service) и балансов/транзакций на стороне эквайера (Bank Provider)                                                           | Блокировка выплат контрагенту, приостановка процессинга и генерация Critical Alert для финансового контроллера            | Ночной Batch-job (Reconciliation), выгружающий логи транзакций из обеих баз и сверяющий нулевую дельту                                           |
| **StuckProcessDetector**        | Выявлять «зависшие» процессы                         | Наличие активных экземпляров процессов (Process Instances) в оркестраторе (Camunda), которые не продвинулись по графу выполнения дольше 1 часа и не породили инцидентов | Оповещение дежурной команды (Alert DevOps) о возможной деградации брокера/воркеров, попытка программного `Retry` инстанса | Мониторинг метрик Zeebe/Operate (Prometheus + Grafana alerts по метрике `zeebe_pending_jobs`) или скрипт-проверка через REST API                 |
